# SimplePortScanner Web Visualization V2 Style Plan

更新时间：2026-07-07

## 1. V2 目标

V2 在 V1 已经完成“本地浏览器可交互扫描”的基础上，尝试把前端改成更有视觉风格和页面层级感的版本。

本方案后续执行时，应优先遵守“用户视觉修订要求”。如果本文档早期描述与修订要求冲突，以修订要求为准。

参考图的方向可以概括为：

- 高对比度。
- 蓝 / 青 / 白 / 黑为主。
- 斜切面板。
- 大字号倾斜菜单。
- 背景有层次，但不喧宾夺主。
- 菜单选项像一个主界面，进入子页面时有明确的切换感。

注意：V2 只参考这种视觉语言，不直接复刻参考图中的角色、商标、具体文字或受版权保护的画面元素。

## 1.1 用户视觉修订要求

当前 V2 初稿不满足预期。后续继续执行时，必须更接近“强视觉海报式菜单”，而不是普通 Web App 或仪表盘。

### 页面构图

- 采用左右分割布局，但不能是简单的 50/50。
- 左侧偏白色，右侧偏深蓝色。
- 中间使用斜切线、三角形、梯形或不规则遮罩，把两个区域切开。
- 整体布局要故意“不完全对齐”，呈现海报排版的动感。
- 页面不要太干净、太对称、太规整。

### 左侧区域

- 左侧可以放一个大面积角色插画、剪影或抽象人物图形。
- 人物不需要完整显示，可以被裁切、旋转、倒置或局部露出。
- 画面边缘可以出现巨大竖排文字，文字可以超出屏幕并被裁掉一部分。
- 左侧主色是白色、黑色、青蓝色，辅以少量粉紫色点缀。
- 可以加入涂鸦线条、手绘笔触、尖锐色块，让画面更有攻击性。

### 右侧区域

- 右侧是主菜单区，用深蓝到亮蓝的渐变作为背景。
- 背景里可以隐约放一张教室、城市、网格、房间或场景图。
- 背景图透明度要很低，只作为氛围，不要抢主体。
- 背景上方覆盖蓝色滤镜，让场景和菜单融合。
- 菜单文字要非常大，不能像普通导航栏一样小。

### 菜单文字

- 只有菜单项使用英文，例如 `SCAN`、`PORTS`、`HELP`、`STATUS`。
- 其余正文、大段说明、提示文字等仍使用中文。
- 菜单项使用粗体、压缩、倾斜的无衬线大写字体。
- 菜单文字应该堆叠排列，略微倾斜，有透视感或错位感。
- 不要使用普通横向 nav bar。
- 菜单项可以竖向堆叠，但不能只是简单竖排；要有堆叠感、倾斜感、错位感。

### 当前选中项

- 当前选中项必须非常突出。
- 推荐用白色倾斜平行四边形作为底板。
- 选中项文字使用黑色粗体。
- 再用一条红色斜杠穿过当前文字，制造“划过”的效果。
- 未选中菜单项使用浅蓝色或青色。
- 未选中项可以带一点阴影、错位描边或发光感。

### 颜色

- 主色是高饱和蓝色和青色。
- 辅助色是白色、黑色。
- 强调色只用少量红色或洋红色。
- 整体颜色应该强烈、干净、锐利。
- 不要灰、不淡、不柔和。

推荐色彩：

```text
深蓝：#0034C8
亮蓝：#007BFF
青色：#54E6FF
浅青：#8DF5FF
白色：#FFFFFF
黑色：#050505
红色强调：#FF1E2D
洋红点缀：#FF2BD6
```

### 形状语言

- 多使用斜线、三角形、平行四边形、锐利切角。
- 少用圆角矩形。
- 按钮不要像普通按钮，而要像贴纸、标签、斜切标题条。
- 可以使用 `z-index` 层叠，让文字、图形、背景互相穿插。

### 排版要求

- 文字可以部分重叠。
- 文字可以轻微旋转。
- 大标题可以被屏幕边缘裁切。
- 菜单区不要太规整，要有视觉节奏。
- 整体要像“菜单正在冲出来”，而不是安静地摆在那里。

### 必须避免

- 不要做成普通仪表盘。
- 不要使用白底卡片列表。
- 不要使用圆角按钮。
- 不要使用细字体。
- 不要使用居中对齐的普通菜单。
- 不要使用柔和 pastel 配色。
- 不要让页面太干净、太对称、太 Web App。

## 2. V2 和 V1 的关系

V1 的后端 API 应尽量保持不变：

```text
GET  /
GET  /styles.css
GET  /app.js
GET  /api/services
POST /api/scan
```

V2 主要改：

- `SimplePortScanner/web/index.html`
- `SimplePortScanner/web/styles.css`
- `SimplePortScanner/web/app.js`

原则：

- 不大规模重构 C++ 扫描核心。
- 不为了 UI 改动后端扫描逻辑。
- 页面层级、布局、动画由前端控制。
- 后端只继续提供数据。

## 3. 推荐页面层级

V2 可以把 V1 的单页工具界面改成“主菜单 + 子页面”的结构。

建议页面状态：

```text
home       主菜单
scan       扫描参数页
scanning   扫描中页
results    扫描结果页
services   常见端口页
help       使用说明页
```

主菜单建议选项：

```text
SCAN
PORTS
HELP
STATUS
```

含义：

- `SCAN`：进入扫描参数页。
- `PORTS`：进入常见端口页。
- `HELP`：进入使用说明页和安全提示。
- `STATUS`：展示当前本地 Web 服务状态、上一次扫描摘要。没有扫描时显示空状态。

扫描完成后自动进入：

```text
results
```

结果页展示开放端口贴纸式条目、斜切结果块和扫描摘要。

## 4. 视觉风格建议

### 4.1 颜色

推荐主色：

```text
背景蓝：#064bd8 / #075cea
高光青：#21e6ff
浅青文字：#9cf7ff
白色面板：#ffffff
黑色强调：#050505
红色强调：#ff2442
```

不要使用过多颜色。V2 应该以蓝、青、白、黑为核心，红色只用于选中、错误或强强调。

### 4.2 构图

推荐结构：

```text
左侧：白色斜切视觉区域，放项目名、状态小框、抽象线条或网络节点
右侧：蓝色半透明主区域，放倾斜菜单或当前子页面内容
背景：淡化的网格、教室/终端/网络拓扑感纹理，不使用复杂真实图片
```

可以用 CSS 实现斜切效果：

```css
clip-path: polygon(...);
transform: skew(...);
```

V2 可以引入人物插画、剪影、抽象人物图形或低透明度氛围背景图。图片只服务构图和氛围，不要抢主菜单和扫描结果主体。CSS 背景、渐变、线条、几何块和文字排版仍然是主要实现手段。

### 4.3 字体与菜单

菜单项建议：

- 大字号。
- 粗体。
- 倾斜。
- 堆叠排列。
- 当前选中项有红色斜切标记或横向划线。

示例方向：

```text
SCAN
PORTS
HELP
STATUS
```

不要让菜单只是普通按钮列表。它应该是 V2 视觉核心。

## 5. 动画原则

动画参考 `docs/advice_for_v2.md` 的核心建议：

不要“为了动画而动画”。不要先强制播放动画，再让用户进入页面。

正确做法：

- 用户点击主菜单后，立即开始进入目标页面状态。
- 如果目标页面需要加载数据或渲染较重内容，则用动画覆盖这段真实等待。
- 数据加载完成后关闭动画并显示子页面。
- 如果加载很快，动画应该有最短显示时间，避免一闪而过。

建议最短动画时间：

```text
250ms - 350ms
```

建议最长感知等待：

```text
不要为了展示动画主动拖慢超过 500ms。
```

## 6. 推荐动画形式

### 6.1 页面进入

点击主菜单进入子页面时：

- 右侧蓝色区域出现斜向 wipe。
- 菜单项向侧边滑出或淡出。
- 子页面标题从右侧或下方切入。
- 内容块、贴纸式条目或斜切标题条分批出现，但总时长保持短。

### 6.2 扫描中

扫描请求真实进行时：

- 显示 `SCANNING` 大字。
- 背景线条或节点轻微移动。
- 显示“等待扫描结果”的状态。
- 不显示假的百分比，除非后端真的提供进度。

V2 仍然不做实时进度流，所以不要显示 `37%` 这类伪进度。

### 6.3 结果页进入

扫描完成进入结果页时：

- 斜切过场。
- 摘要数字快速浮入。
- 开放端口贴纸式条目或斜切结果块依次轻微上浮。

如果没有开放端口：

- 显示明确空状态。
- 不要把空结果做成错误。

## 7. 前端实现建议

V2 仍可以不用 React / Vue。用原生 JS 做一个轻量状态机即可。

建议结构：

```js
const state = {
  view: "home",
  services: [],
  lastSummary: null,
  lastResults: []
};

function navigate(view, options = {}) {}
function render() {}
function renderHome() {}
function renderScan() {}
function renderScanning() {}
function renderResults() {}
function renderServices() {}
function renderHelp() {}
```

动画可以封装成：

```js
async function withPageTransition(loadPage) {
  openTransition();
  const start = performance.now();

  await loadPage();

  const elapsed = performance.now() - start;
  if (elapsed < 300) {
    await delay(300 - elapsed);
  }

  closeTransition();
}
```

这和 `advice_for_v2.md` 的思路一致：动画覆盖真实加载过程，并设置最短时间避免闪烁。

## 8. 子页面交互

### 8.1 扫描参数页

保留 V1 表单能力：

- IP / IP 范围。
- 端口表达式。
- 超时时间。
- 线程数。
- 快捷端口。
- 开始扫描。

点击开始扫描后：

1. 切到 `scanning` 状态。
2. 发起 `POST /api/scan`。
3. 请求完成后切到 `results` 状态。

### 8.2 结果页

结果页展示：

- 摘要数字。
- 开放端口贴纸式条目或斜切结果块。
- Banner。
- 返回主菜单。
- 返回扫描参数页。

结果页不要使用传统数据网格作为主视觉。

推荐结果条目：

```text
OPEN
127.0.0.1:8080
HTTP-Proxy
12 ms
Banner: ...
```

### 8.3 常见端口页

展示 `GET /api/services` 返回的数据。

表现形式可以是：

- 端口芯片。
- 斜切列表。
- 分组贴纸或斜切列表。

### 8.4 帮助页

展示：

- IP 输入说明。
- 端口输入说明。
- 授权扫描提示。
- 本地 Web 服务说明。

## 9. 可访问性和降级

需要支持用户系统的减少动画设置：

```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

如果浏览器不支持复杂 `clip-path`，页面仍应可用，只是视觉效果弱一些。

## 10. V2 不做的事

V2 不做：

- 后端大重构。
- 扫描核心重写。
- 假进度百分比。
- 扫描历史。
- 文件导出。
- 远程访问。
- 登录系统。
- 大型前端框架迁移。

## 11. V2 验收标准

功能：

- 主菜单能进入扫描、常见端口、帮助、状态页面。
- 扫描流程仍然可用。
- 扫描完成后进入结果页。
- 结果页能展示摘要和开放端口贴纸式条目。
- 无开放端口时有空状态。
- 原后端 API 不因视觉改造而大改。

视觉：

- 有明确的蓝 / 青 / 白 / 黑高对比风格。
- 有斜切构图和大字号菜单。
- 子页面切换有短促、清晰、不拖慢用户的动画。
- 结果页不是传统数据网格风格。

工程：

- 主要改动集中在 `web/`。
- C++ 后端只在必要时做小修。
- V1 功能不退化。

## 12. 结论

V2 可以大胆改页面架构和跳转方式，但要保持后端 API 稳定。动画应该服务于真实页面切换和数据加载，不能变成阻塞用户操作的装饰。

推荐方向：

```text
主菜单式高对比视觉 + 子页面状态机 + 真实加载过场动画 + 贴纸式结果条目
```
